Projects listed here are a curated subset of projects I work on. See more at GitHub.

Audlibri

  • Multi-platform audiobook application written in HTML, JavaScript, Java, and C#
  • Idea came together after reading over the Chrome extention APIs released by Google in December 2010.
  • First version of Chrome extension/app written in one night, Christmas Eve 2010.
  • Chrome app has over 3,500 users today.
  • Chrome app utilized the then new HTML5 Audio/Video tag for playing audio in browser. The audio tag didn't include a way to keep a playlist of tracks to be played in order, so I wrote and open-sourced Phonograph.js in CoffeeScript for managing playlists.
  • The Chrome app pulled from the Librivox API, so to create a smoother experience I wrote my own backend and API. Each book is stored in a MongoDB database. Originally the API was written to coincide with the website for Audlibri but the Chrome app was later rewritten to use the API as well.
  • I began working on the Android app in February 2011 and released the first version to the Android market in June 2011. The Android application uses background threads to call the API for searching/browsing audiobooks, and uses a Service to play audiobooks when the user is not interacting withe app directly.

Atwork.mobi

  • Atwork.mobi is a mobile application written in html and JavaScript for consultants to keep track of the time spent working on client projects.
  • Fully open-sourced from the beginning, source available on GitHub.
  • Uses LocalStorage to maintain state. While the timer is running the user can change to another tab, look at their work log, and even close the site entirely but the timer will continue to run.
  • Uses an IndexedDB (which has sketchy support in mobile browsers) to store sessions the consultant has ended. Supports both the old version of IndexedDB (still being used by WebKit browsers) and the new version (as of now only supported by Firefox).
  • Uses CSS3 transitions for a unique tab-switching effect when moving between the work and log tabs.
  • Written entirely in custom JavaScript, without any bulky third-party libraries such as JQuery so the application is as responsive as any native counterpart in the same category.
  • Using Application Cache to bring downtime to a minimum. Over 1 weekend the site went down due to a botched upgrade (I had to wait for my VPS to reset my ssh public key) but the application continued to work for people who already had it "installed" on their phones. Application cache also reduces the amount of bandwidth my small VPS consumes, on page refreshes only the Manifest.appcache file (small plain text file) is downloaded.

Finger.js

  • Finger.js is a finger server (source code) written for Node.js.
  • Finger.js currently supports a subset of the Finger protocol Q1 specification, excluding those features which have shown to be a security risk in the years since finger was originally created.
  • Can run daemonized as a service on startup and receives requests on port 79.
  • Will read from a user's .plan or .project files and return the contents to the requestor. Supported both unix (including Mac OSX) and Windows as its host server

Maps

  • My (unnamed) mapping application was created due to my frustration of current html-based mapping software on mobile phones. All of the major mapping providers (Google Maps, Yahoo Maps, and Mapquest) use user-agent sniffing to prevent browsers other than Mobile Safari and Android browser from visiting their sites. Additionally these sites, even when rendered in those providers, only provide a small subset of the functionality than the destkop versions provide.
  • Uses the terrific Leaflet project for rendering the map layer and Mapquest as the provider for Open Street Maps data.
